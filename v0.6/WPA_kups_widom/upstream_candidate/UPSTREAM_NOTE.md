> validator. The model is the MACE team's, the engine is CuspAI's; this is help, not criticism.

## TL;DR
To our knowledge, based on public-source searches conducted on 15 June 2026, we ran the first public independent cross-engine numerical validation of kUPS's Widom test-particle module (source
`main`; not in released `kups 1.0.1`). On two locked positive controls it reproduces an independent
estimator and a RASPA3 reference **to within statistical error on q_st, and on K_H once one LJ
truncation convention is matched**. The JAX MLIP export is energy-identical to upstream torch MACE. One
small documentation suggestion follows; everything else is a clean bill of health.

## What was validated

### 1. Henry coefficient & heat of adsorption — two controls
| control | observable | kUPS | independent ref | agreement |
|---|---|---|---|---|
| **6c** MFI + Ar (LJ only, 12 Å) | q_st | 16.25 kJ/mol | 15.9 native / 16.0 RASPA3 | **0.2 kJ/mol** |
| | K_H | 0.243 mol·kg⁻¹·bar⁻¹ | 0.207 RASPA3 | +16 % → see §2 |
| **4c** Si-CHA + CO₂ (TraPPE, charges, **Ewald**, 14 Å) | q_st | 22.98 kJ/mol | 22.99 native | **0.01 kJ/mol** |
| | K_H | 2.271 mol·kg⁻¹·bar⁻¹ | 2.215 native (truncated → 2.472) | see §2 (convention) |

The estimator's physics is correct: W = exp(−βU), and μ_ex / K_H / q_st reduce exactly as the textbook
Widom definitions; **q_st matches to 0.01 kJ/mol** so the electrostatic energies (CO₂ + framework charges
+ Ewald) are right. The K_H comparison must be read on a matched truncation convention — see §2.

### 2. The one actionable point — `lennard_jones_energy` is truncated, not energy-shifted
The only non-trivial discrepancy (the 6c +16 % on K_H) root-causes cleanly to an LJ **truncation
convention**. kUPS's `lennard_jones_energy` masks the pair energy at `r < cutoff` with **no `−U(r_c)`
shift**; the RASPA3 recipe we compared against uses `TruncationMethod: shifted`. The unshifted potential
is more attractive by `U(r_c)` per interaction (here a near-constant ≈0.43 kJ/mol per insertion), which
raises ⟨exp(−βU)⟩ by exp(βδ) ≈ 1.16 — exactly the +16 %. Toggling the shift off in the independent
estimator reproduces kUPS to **+2.4 % on K_H and 0.09 kJ/mol on q_st**, both within the kUPS SEM. The
shift is still substantial at the 4c cutoff (14 Å): a fresh native convention matrix gives shift =
**+11.6 %** there (and tail = +11.9 %). kUPS **does** apply a tail in the Widom path (we tested it —
`tail_correction: false` drops K_H −16 %), so the matched cell is truncated **+tail**: read that way,
kUPS is **−8.1 %** vs native (−11.7 % tail-off-both). The residual is **q_st-invariant** (Δq_st
≤ 0.3 kJ/mol) → a normalization-flavoured ⟨W⟩ scale, with an identified ~4 % piece from a tail-magnitude
difference (kUPS +16.3 % vs native +11.9 %) and the rest from the Ewald convention / CO₂ sampling /
volume-or-count normalization. Flagged honestly rather than papered over by the coincidental +2.2 %
against the shifted reference; q_st matches to 0.01 kJ/mol, so it is not the energies.

**This is not a bug** — truncated-not-shifted is a legitimate choice. The suggestion is purely
documentation: a one-line note in the LJ docstring / config schema that the default is *truncated
(unshifted)*, so users transferring a force field from a shifted-convention engine (RASPA, and most
GCMC codes) know to expect a ~cutoff-dependent K_H difference unless they match the convention. An
optional `shift: true` flag (subtract `U(r_c)`) would make round-tripping RASPA recipes exact.

### 3. MLIP export — faithful
kUPS's `TojaxedMliap` JAX export of MACE-MPA-0 is **energy-identical to upstream torch MACE-MPA-0 to a
max relative difference of 2.1×10⁻⁷** across test geometries — independently confirming the model card's
stated rtol = 1e-4.

## Validation fixture offered
`widom_validation_6c.yaml` (Ar/MFI, seed 0) and the 4c CO₂/Ewald config are ready-to-run regression
fixtures with reference values inline — useful as CI for the Widom module. Happy to contribute the full
cross-engine harness.

## A recipe-completeness note (general, not kUPS-specific)
The validation also re-surfaced a transfer hazard worth a doc line for anyone porting force fields into
kUPS: on this all-silica control the **Ar–Si interaction generated by Lorentz-Berthelot from the
framework self-parameters is load-bearing** — dropping it (keeping only the explicit Ar–O cross-pair)
under-predicts K_H by 1.8×. kUPS handles this correctly when the framework self-parameters are supplied;
the note is just that they *must* be. Likewise the `blocking_spheres` API is the right place to encode
short-range protection, and (separately) the MACE/UMA potentials carry no dispersion term, so adsorption
users should pair them with a dispersion correction — all consistent with kUPS's design.
